## 1. Vue d'ensemble

AGRAFES est un concordancier de corpus multilingues sous forme d'application de bureau
Tauri 2.0. Il articule :

- un **moteur Python** (`multicorpus_engine`) exposé comme sidecar HTTP persistant
- trois **frontends Tauri** : `tauri-app` (concordancier), `tauri-prep` (constitution),
  `tauri-shell` (coquille unifiée)

Le projet est à un stade **beta fonctionnel** : les fonctionnalités cœur sont toutes
implémentées, mais plusieurs couches de finition et de robustesse manquent.

## 2. Forces

### 2.1 Architecture backend solide

| Point fort | Détail |
|------------|--------|
| Zéro dépendance framework | `http.server` stdlib, pas de Flask/FastAPI → déploiement trivial, surface d'attaque minimale |
| Contrat API versionnée | `sidecar_contract.py` + `SIDECAR_API_CONTRACT.md` → évolutions sans casse |
| Migrations SQL ordonnées | 11 migrations forward-only, numérotées, jouées au démarrage |
| Couverture de tests backend | 47 fichiers pytest couvrent API, importers, exporters, alignement, TEI |
| Logging reproductible | Table `runs` : chaque opération est journalisée avec ses paramètres |
| Formats d'import larges | 9 modes (DOCX/ODT lignes numérotées, paragraphes, TXT, TEI) |
| Exports riches | 7 formats : JSONL, CSV (plat/long/famille), TEI, TEI-ZIP, TMX, bilingue, texte lisible |

### 2.2 Frontend TypeScript propre

| Point fort | Détail |
|------------|--------|
| TypeScript strict | `strict: true` dans tsconfig, peu de `any` résiduels |
| Pattern DI manuel | Les imports circulaires (query ↔ results) sont résolus par injection de callback |
| Virtualisation des résultats | `content-visibility: auto` + IntersectionObserver + plafond 150 nœuds DOM |
| Pagination lazy | Sentinel IntersectionObserver, "Charger plus" fiable |
| Deep-link `agrafes://` | Ouverture de corpus depuis l'explorateur de fichiers |
| Portfile de découverte sidecar | Le sidecar déjà actif est réutilisé sans relance |

### 2.3 CI/CD & distribution

- Build multiplateforme (Windows NSIS, macOS DMG notarisé, Linux AppImage)
- Signature Authenticode Windows + notarisation macOS
- 11 workflows GitHub Actions couvrent build, test, release, signing

## 3. Faiblesses

### 3.1 Critiques (bloquent la fiabilité en production)

#### W-01 — `shell.ts` est un monolithe de 117 000 lignes
`tauri-shell/src/shell.ts` concentre le routeur, le header, la gestion DB,
le cycle de vie des modules et des dizaines de gestionnaires d'événements dans
un seul fichier. Ce n'est pas maintenable :
- impossible de tester unitairement
- diff git illisible
- temps de compilation TypeScript significatif
- risque élevé de régression lors de tout ajout

#### W-02 — `tauri-prep/src/app.ts` monolithique (~40 000 lignes estimées)
Même problème côté Constituer : un seul fichier pour 7 écrans distincts.
Aucune séparation logique entre les écrans Alignment, Segmentation, Export, Audit, etc.

#### W-03 — Absence totale de tests frontend
Aucun test unitaire JS/TS, aucun test E2E (Playwright/WebdriverIO).
Seuls des smoke tests manuels en CI. Toute régression sur les écrans passe inaperçue
jusqu'à la revue manuelle.

#### W-04 — État global mutable sans observateurs
Le singleton `state` est muté directement (`state.filterLang = value`).
Les re-rendus sont déclenchés par appels manuels dispersés dans le code.
Un oubli de `renderChips()` ou `renderResults()` produit un UI désynchronisé silencieusement.

#### W-05 — Pas de rollback de migration SQL
Les migrations sont forward-only. Si une migration 012 introduit un bug de schéma,
il n'y a aucun chemin de retour automatique. Le seul recours est de restaurer la DB.

### 3.2 Importantes (dégradent l'expérience ou la qualité)

#### W-06 — Accessibilité absente
Pas de labels ARIA sur les boutons icônes ("⚙", "?", "⬆"), pas de rôles
`role="dialog"` sur les modales, pas de focus-trap dans les panneaux.
La navigation clavier est partielle (Enter sur la barre de recherche, Escape
sur certains panneaux seulement).

#### W-07 — Sidecar à connexion SQLite unique avec verrou global
`threading.Lock` protège toutes les écritures, mais une longue exportation
(TEI, TMX) bloque la connexion pendant toute la durée du job. Les requêtes de
lecture concurrent sont sérialisées. Sur un corpus volumineux, l'UI peut paraître
gelée (le spinner tourne, mais les réponses /query attendent le verrou).

#### W-08 — Pas de gestion d'erreur réseau côté frontend
Si le sidecar se termine inopinément, `sidecarClient.ts` lève une exception
non catchée dans certains flux. L'UI s'arrête sans message explicatif.
Quelques `try/catch` sont présents, mais la couverture est inégale.

#### W-09 — Historique de recherche localStorage uniquement
Pas de portabilité entre machines, pas de persistence par DB (si on change de corpus,
l'historique est le même pour tous). Risque de collision si on ouvre deux instances
sur des DB différentes dans le même profil.

#### W-10 — CSS en fichiers multiples non bundlés dans tauri-shell
`styleRegistry.ts` charge dynamiquement des feuilles CSS depuis les modules
(explorer, constituer). Cela fonctionne en dev mais peut poser des problèmes
de FOUC (flash of unstyled content) et de priorité de cascade en production.

### 3.3 Mineures (dette technique acceptable à court terme)

- **W-11** — Pas de `.env` / configuration externalisée (chemins hardcodés dans certains tests)
- **W-12** — `type-ignore` dans `sidecar.py` (10 occurrences) — devraient être typés proprement
- **W-13** — `multicorpus serve` ne supporte pas le rechargement à chaud du token (redémarrage nécessaire)
- **W-14** — Pas de rate-limiting sur le sidecar HTTP (acceptable en local, problématique si exposé)
- **W-15** — Pas de versionnement des documents (une seule version par doc, pas d'historique)

## 4. Branchements des écrans — état des connexions

### 4.1 `tauri-shell` (coquille unifiée)

```
[Démarrage] ──► initShell()
                 │
                 ▼
              [Header fixe 44px]
              ├── DB Zone: "Ouvrir…" / "Créer…"  ──► sidecar spawn + portfile
              └── Tabs: [Explorer] [Constituer]
                         │              │
                         ▼              ▼
                  explorerModule  constituerModule
                  (mount/dispose) (mount/dispose)
```

**Verdict :** ✅ Le routeur shell fonctionne. La persistance `lastMode`/`lastDbPath`
via localStorage est opérationnelle. Le cycle `dispose()` avant navigation est présent.

**Problème détecté :** si le sidecar ne répond pas au moment du `mount()` de l'Explorer,
il n'y a pas de retry avec back-off exponentiel — seulement un timeout fixe.

---

### 4.2 `tauri-app` (Concordancier / Explorer)

```
[Bootstrap]
  │  détecter portfile ou lancer sidecar
  ▼
[Écran principal — single page]
  │
  ├── [Barre de recherche] ──► query.ts → POST /query → renderResults()
  │
  ├── [Tiroir Filtres] ──────► filters.ts → GET /families, /documents
  │   ├── lang / role / resource_type / family dropdowns
  │   └── DocSelector (multi-select persisté)
  │
  ├── [Panneau Requête] ──────► search.ts → génère FTS5 query string
  │
  ├── [Historique] ────────────► history.ts → localStorage read/write
  │
  ├── [Import modal] ──────────► importFlow.ts → POST /import
  │   └── Après import : GET /documents, re-render filters
  │
  ├── [Export menu] ───────────► export.ts → POST /export/csv ou GET /query (JSONL)
  │
  ├── [Panneau Méta] ──────────► metaPanel.ts → GET /documents/{id}
  │
  └── [Résultats cards]
       ├── "Charger plus" ──────► nextOffset → POST /query (append)
       ├── "Filtre par doc" ────► state.filterDocIds push + re-query
       ├── "Copier" ────────────► navigator.clipboard (sync, pas de feedback d'erreur)
       └── "Meta" ──────────────► metaPanel.ts (ouvre panneau latéral)
```

**Verdict :** ✅ Tous les flux principaux sont connectés et fonctionnels.

**Problèmes détectés :**
- Le bouton "Copier" n'affiche pas de feedback si `navigator.clipboard` échoue
  (API indisponible dans certains contextes Tauri sans permission explicite).
- "Réinitialiser" remet à zéro l'état mais **ne ferme pas** les panneaux ouverts
  (tiroir Filtres, panneau Requête) — l'UI reste dans un état visuellement incohérent.
- Après un import réussi, le filtre famille n'est pas rafraîchi automatiquement
  si le document importé appartient à une nouvelle famille.

### 4.3 `tauri-prep` (Constituer)

```
[Home/Upload]
  │  drag-and-drop ou sélection fichier
  ▼
[Documents list]
  ├── Edition métadonnées (inline)
  ├── Hiérarchie (famille parent/enfant)
  └── Actions : Supprimer, Segmenter, Aligner
       │
       ▼
[Alignment screen]
  ├── Sélection paire (parent + enfant)
  └── Choix stratégie (external_id / position / similarity)
       │
       ▼
[Segmentation screen]
  ├── Sélection document + langue pack
  └── POST /segment (async job)
       │
       ▼
[Export screen]
  ├── TEI / TEI-ZIP / TMX / Bilingue / Texte lisible
  └── POST /export/* → GET /jobs/{id} (polling)
       │
       ▼
[Job Center]
  └── Liste des jobs en cours / terminés / en erreur
       │
       ▼
[Audit screen]
  └── GET /corpus/audit → tableau de bord qualité
```

**Verdict :** ⚠️ La chaîne logique est cohérente mais les transitions entre écrans
sont toutes dans `app.ts` sans routeur explicite. L'ordre "idéal" (Upload → Align →
Segment → Export) n'est pas guidé — un utilisateur peut lancer un export sans avoir
segmenté ni aligné.

**Problèmes détectés :**
- Pas de guard de navigation ("Avez-vous segmenté avant d'exporter ?")
- L'écran Audit est accessible depuis le menu mais son lien retour vers les actions
  correctives (ex : cliquer sur un warning → ouvrir le document concerné) n'est pas implémenté.
- Le Job Center ne distingue pas visuellement les jobs d'export des jobs de segmentation
  (même icône, même couleur).

### 4.4 Connexion Shell ↔ Modules

```
Shell (tauri-shell)
  │
  ├── explorerModule.ts
  │    mount()  ──► injecte le DOM de tauri-app dans #module-container
  │    dispose() ─► retire les écouteurs, arrête le polling
  │
  └── constituerModule.ts
       mount()  ──► injecte le DOM de tauri-prep dans #module-container
       dispose() ─► idem
```

**Verdict :** ⚠️ Le pattern mount/dispose est fonctionnel mais fragile.

**Problèmes détectés :**
- Les deux modules partagent le même sidecar (même `dbPath`) mais chacun maintient
  sa propre instance de `sidecarClient`. Si une opération longue est en cours dans
  Constituer et que l'utilisateur bascule sur Explorer, les deux clients émettent
  des requêtes vers le même sidecar — pas de coordination.
- Il n'y a pas de confirmation avant de changer de module si un job est en cours
  dans Constituer (risque de perte de contexte).
- `styleRegistry.ts` charge les CSS dynamiquement ; si un module est monté deux fois
  de suite (retour rapide), les styles risquent d'être dupliqués dans le DOM.
