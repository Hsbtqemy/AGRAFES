## 1. Synthèse exécutive

Depuis l'audit du 2026-06-12, **la quasi-totalité des chantiers structurels a été menée à bien
et c'est vérifié dans le code** (pas seulement dans le tracker) : couche `services/` extraite
(A-01 point d'arrêt), validation déclarative (A-03/A-03B), `ruff` vert sur `src`+`tests` (Q-01),
gate de couverture `--cov-fail-under=60` (T-01), `curation.py` à 100 % (T-02), dependabot étendu
à pip/npm/cargo (N-01), versions shell+engine synchronisées à 0.2.7 (N-07), client sidecar unifié
dans `shared/sidecarCore.ts` (U-01), durcissements sécurité S-01/S-02/S-03/S-04 et la série
runtime R-01a→d. **Le socle de gouvernance reste exemplaire pour un développement solo.**

Mais cette passe — plus fine et plus orientée *correction* que les précédentes (qui étaient
surtout architecturales) — a fait remonter **un cluster de bugs réels que les audits antérieurs
n'avaient pas vus**, souvent parce qu'ils échappent aux tests existants :

1. **Deux bugs 🔴 de correction** confirmés manuellement : un **DoS algorithmique CQL** reproduit
   en 114 s (QRY-01), et un endpoint GET qui renvoie **500 au lieu de 400** sur une erreur de
   saisie à cause d'un appel `_send_error` positionnel qui lève `TypeError` (SID-01).
2. **Un cluster sécurité 🟠 local mais exploitable** : `/jobs` POST mute la DB **sans token**
   (bypass d'auth, SID-02/SEC-01), injection de formule **CSV** à l'export (QRY-02), **XSS** dans
   le rapport QA HTML (QRY-03), **SSRF** WebDAV non bloqué vers le réseau interne (SID-04), couleur
   de convention injectée sans `safeColor` ni validation serveur (SEC-02/FE-01).
3. **Des trous d'intégrité** : mutations multi-statements **non atomiques sans rollback** sur la
   connexion partagée (SID-03), provenance ShareDocs corruptible par crash entre deux commits
   (SID-05), **perte silencieuse de la mise en forme à l'import ODT** par un mauvais ordre
   d'arguments (ENG-01).
4. **Des régressions de filet/gouvernance** : le growth-gate `sidecar.py` est devenu **non
   bloquant** (`exit 0`, OPS-01), `smoke.yml` **ne tire pas sur `dev`** (la branche PR par défaut,
   OPS-02), et le drift `API_VERSION` (1.6.23) vs `CONTRACT_VERSION` (1.6.31) **s'est creusé** à
   8 patches (OPS-03).

Aucune vulnérabilité critique exploitable à distance (le sidecar reste loopback-only, SQL
intégralement paramétré, pas d'`eval`/`exec`/`pickle`). Le modèle de menace dominant est : *fichier
de corpus / fichier WebDAV piégé* → données non fiables qui ressortent en export/HTML, et *process
local même-utilisateur* → bypass du token loopback.

## 3. Forces vérifiées (échantillon contrôlé dans le code)

**Moteur / données.** Tous les index DB présents (`idx_alinks_*` 003/008, `idx_tokens_*` 012,
`idx_units_*` 001/014) — A-05 reste réfuté. Imports atomiques (transaction unique par fichier +
`try/except: rollback; raise`, CoNLL-U inclus). Suppression de document intègre (DELETE explicites
+ cascades FK, `foreign_keys=ON`). Migrations sous `_migration_lock`, idempotentes, tri numérique,
support PyInstaller. `normalize()` cohérent import↔index (mêmes valeurs réutilisées, pas de
re-normalisation divergente). Dedup `source_hash` à 4 niveaux, aligné front (long-path Windows
géré). WAL + `synchronous=FULL` + `foreign_keys=ON`. **ADR-043 `text_source` propagé** correctement
à l'import/resegment/merge/split/undo (N-04 clos-vérifié).

**Requête / export.** SQL 100 % paramétré ; les seules interpolations f-string sont des IN-lists
`?,?,?` et des noms de colonnes **validés par allowlist** (`token_collocates`, `token_stats`). Toutes
les divisions de stats gardées (PMI, G², freq_per_10k, ratio→inf neutralisé avant JSON).
`html_export.py` échappe tout via `html.escape`. `exporters/tei.py` **write-only** (0 `.parse()`,
S-04 clos). Exporters CoNLL-U/SKE neutralisent tabs/attributs. Encodage export UTF-8 forcé.

**Sidecar / concurrence.** **R-01b vérifié sans race** : tout le dispatch DB est sous `RLock`,
carve-outs lock-free (`/health`, `/openapi.json`, `/shutdown`, `/webdav/list`) tous prouvés DB-free,
runner async mute sous lock, query fédérée ferme `ext_conn` en `finally`. Token `hmac.compare_digest`
+ garde anti-amplification. `_check_host` (DNS-rebinding) et portfile `O_EXCL|0o600` corrects.
Couche services : protocole `(conn, …)` + erreurs typées respecté, 7/7 services testés.
Contract-freeze opérationnel (85 paths, aucun endpoint au contrat sans implémentation).

**Sécurité WebDAV.** `defusedxml` partout (pas de XXE), drop des `href` off-origin avant tout GET
porteur d'auth, strip `Authorization` sur redirect cross-origin, `validate_remote_url` bloque
`file://`, cap de taille triple-gardé (PROPFIND + Content-Length + streaming), credentials en
keychain OS natif, jamais en DB/logs/réponse/`runs.params`.

**Front / Rust.** Client sidecar **réellement unifié** (un seul `_conn` dans `shared/sidecarCore.ts`,
le shell ne bundle qu'un cœur — U-01 clos, le « dual state » est un faux positif). Sink `safeHtml`
centralisé, ESLint `no-unsanitized` à `error` sur prep/app/shell, aucune baseline tolérée. Spawn
sidecar non-injectable (argv séparé, host/port en dur), allowlist HTTP loopback-only, portfile
whitelisté au nom exact (C-01 clos), update-check GitHub validé `[a-zA-Z0-9._-]` (ADR-039).

**Process.** `permissions:` durci sur les 14 workflows (`contents: write` limité aux 4 de release),
aucun `pull_request_target`, contract-freeze + lint XSS + vitest gatés, dependabot 4 écosystèmes,
`bump_version.py` couvre les 6 vecteurs de version.

## 7. Conclusion

Le projet a **réellement encaissé** les chantiers structurels des audits précédents : la couche
services, le filet CI (lint + couverture + dependabot), l'unification du client sidecar, les
durcissements sécurité et la série runtime R-01 sont en place et vérifiés dans le code. La
gouvernance solo reste au-dessus de la moyenne.

L'apport propre de cette passe est d'avoir creusé **sous l'architecture, au niveau de la
correction** : elle remonte deux bugs 🔴 (DoS CQL, mauvais code HTTP), un cluster sécurité 🟠 *local
mais exploitable* (token `/jobs`, injection CSV, XSS QA, SSRF, couleur de convention), et des trous
d'intégrité (mutations non atomiques sans rollback, provenance ShareDocs, perte de mise en forme
ODT). Tous ont **traversé les tests existants** — le signal de fond est que la couverture, bien que
chiffrée à 60 %+, ne touche pas encore les chemins d'erreur, les données piégées et les cas limites.

La pile des chantiers se hiérarchise ainsi : **P0 = la poignée de correctifs ponctuels ci-dessus
(1-2 jours)**, puis durcissement sécurité/intégrité (P1), puis le reste de la dette connue et
assumée (A-01 sidecar, U-02 écrans, F-03 SHA-pinning, E2E). Aucun de ces points ne remet en cause
la santé d'ensemble ; ils relèvent d'un produit qui est passé du « bien architecturé » au « à
durcir dans le détail ».

— *Audit produit par 6 agents d'analyse parallèles + contre-vérification manuelle des findings
tête d'affiche. Aucune modification de code effectuée.*
